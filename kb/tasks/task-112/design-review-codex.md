## Design Review: task-112 — 2026-07-12
- Reviewer: Codex
- Review status: request-changes
- Feedback:
  - 범위/하드캡은 대체로 승인한다. task-112는 이벤트 4종만 포함하고(`kb/tasks/task-112/design.md:39`, `kb/tasks/task-112/design.md:44`), 5번째 이벤트·신규 씬·신규 매니저·저장/불러오기를 제외한다(`kb/tasks/task-112/design.md:63`, `kb/tasks/task-112/design.md:67`, `kb/tasks/task-112/design.md:70`). 이는 demo-scope의 이벤트 4종, 씬 2개 하드캡과 정합한다(`kb/concepts/demo-scope.md:21`, `kb/concepts/demo-scope.md:33`, `kb/concepts/demo-scope.md:35`).
  - 결정론 계약은 task-110/111과 정합한다. FNV-1a 재사용, strict `<` 문턱, 시드 47 known vector, milli 정수 투영, Day 1 보호, 예고==적용 테스트가 명시되어 있다(`kb/tasks/task-112/design.md:214`, `kb/tasks/task-112/design.md:220`, `kb/tasks/task-112/design.md:227`, `kb/tasks/task-112/design.md:325`, `kb/tasks/task-112/design.md:827`, `kb/tasks/task-112/design.md:839`). task-111의 DayModifier/neutral overload/base-prefix 계약도 보존된다(`kb/tasks/task-111/design.md:245`, `kb/tasks/task-111/design.md:265`, `kb/tasks/task-111/design.md:286`).
  - 핵심 설계 결정인 `EventDayEffects`와 `DayModifier` 분리는 승인한다. 재료값/운영비는 경제 축 DTO로, 단체 주문은 `TryComposeDayModifier`를 거쳐 수요 축으로 전달하는 구조가 Economy/Settlement를 SNS 고객 투영 입력에 결합하지 않게 한다(`kb/tasks/task-112/design.md:346`, `kb/tasks/task-112/design.md:373`, `kb/tasks/task-112/design.md:387`). 두 DTO 모두 `activeEvents`+defs에서 `EventOps` 단일 원천으로 파생된다는 의존 그림도 명확하다(`kb/tasks/task-112/design.md:560`, `kb/tasks/task-112/design.md:563`, `kb/tasks/task-112/design.md:568`). `EventDayEffects`가 단체 필드를 중간 projection으로 포함하는 점은 허용 가능하다. 최종 수요 소비자가 `DayModifier/GenreDemandPlan`으로 제한되기 때문이다.
  - task-111 리뷰 001의 교훈은 반영되어 있다. 모든 def 참조 경로의 공통 검증을 요구하고(`kb/tasks/task-112/design.md:264`), 손상 상태의 명시적 실패와 상태 불변을 테스트 기준으로 둔다(`kb/tasks/task-112/design.md:834`). JsonUtility 왕복 후 `activeEvents`와 plan 동등성까지 포함한다(`kb/tasks/task-112/design.md:868`). 이는 task-111 리뷰의 누락 지점과 직접 대응한다(`kb/tasks/task-111/reviews/001.md:8`, `kb/tasks/task-111/reviews/001.md:9`).
  - 임대료 `durationDays=0` 영구 규약은 건전하다. `ActiveEventState.remainingDays=0`을 영구 상태로 미러링하고(`kb/tasks/task-112/design.md:190`, `kb/tasks/task-112/design.md:194`), 수명 전이와 run당 1회 발생 보장을 테스트한다(`kb/tasks/task-112/design.md:250`, `kb/tasks/task-112/design.md:832`).
  - 기존 API 하위호환과 기준선 무회귀는 반영되어 있다. neutral overload 위임과 task-111 기준선 EditMode 236/PlayMode 4 보존이 제약과 테스트 기준에 들어 있다(`kb/tasks/task-112/design.md:104`, `kb/tasks/task-112/design.md:823`, `kb/tasks/task-112/design.md:845`, `kb/tasks/task-112/design.md:894`). day >= 3 기대값 변화도 의도된 변화로 식별·기록하도록 되어 있다(`kb/tasks/task-112/design.md:105`, `kb/tasks/task-112/design.md:702`, `kb/tasks/task-112/design.md:810`, `kb/tasks/task-112/design.md:932`).
  - 변경 요청: 정산 원인 라인의 구매 할증 입력 계약이 한 단계 모호하다. `BuildSettlementCauseLine` API는 `eventSurcharge`만 받지만(`kb/tasks/task-112/design.md:291`), F5는 `marketSpendDay == day`일 때의 추적값을 사용한다고 정의한다(`kb/tasks/task-112/design.md:654`). 이 필터가 API 계약에 없으면 호출자가 실수로 이전 day의 `marketEventSurchargeToday`를 전달해도 문자열 조립 단일 원천이 막지 못한다. 리셋 테스트는 있지만(`kb/tasks/task-112/design.md:860`), `marketSpendDay != day`인 stale surcharge가 원인 라인에 표시되지 않는다는 직접 기준은 없다.
  - 변경 요청: 정산 원인 라인의 최악 길이 대응이 구현 전 계약으로 닫히지 않았다. F5는 460x14 한 줄을 계약하고(`kb/tasks/task-112/design.md:641`, `kb/tasks/task-112/design.md:647`), 오픈 이슈는 4종 동시 활성 시 460px 초과 가능성을 인정한 뒤 화면 리뷰로 미룬다(`kb/tasks/task-112/design.md:921`). 하지만 이 라인은 구현자가 지금 배치해야 하는 UI이므로, 다중 이벤트 시 축약 규칙 또는 2줄 fallback을 설계에 고정해야 한다. 현재 테스트 기준은 좌표/겹침 확인과 수동 시각 승인에 의존한다(`kb/tasks/task-112/design.md:877`, `kb/tasks/task-112/design.md:898`).
  - 참고: `python -B runtime/validator/cli.py kb/tasks/task-112/design.md`는 통과했다.
- Action required:
  - `BuildSettlementCauseLine`의 입력 계약을 명확히 하라. 권장: `day`와 `marketSpendDay`를 함께 받거나, 호출자가 `marketSpendDay == day ? marketEventSurchargeToday : 0`으로 필터링한 값만 전달한다고 C3/F5/API 주석에 고정하고 stale-day 테스트를 추가한다.
  - F5 정산 원인 라인의 다중 이벤트 fallback을 구현 전 확정하라. 권장: 3종 이상 활성 시 축약 포맷을 쓰거나 2줄 TMP/MessageText 재배치를 설계하고, worst-case 문자열이 460px 안에 들어가거나 겹치지 않는 테스트 기준을 추가한다.
